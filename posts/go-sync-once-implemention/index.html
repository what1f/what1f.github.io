sync.Once

type Once struct {
	// done indicates whether the action has been performed.
	// It is first in the struct because it is used in the hot path.
	// The hot path is inlined at every call site.
	// Placing done first allows more compact instructions on some architectures (amd64/386),
	// and fewer instructions (to calculate offset) on other architectures.
	done uint32
	m    Mutex
}

Once的结构比较简单，内部只有done和Mutex两个变量，done用来表示调用是否完成，初始值为0，如果成功执行过一次Do时，done值变为1。

这段注释很有意思，"It is first in the struct because it is used in the hot path." 大概意思就是done这个变量位于struct的第一个值是因为done被使用“热路径”(hot path)里，我的理解就是在Once这个结构中，done变量使用得更频繁。"Placing done first allows more compact instructions on some architectures (amd64/386),and fewer instructions (to calculate offset) on other architectures." 意思是把done放在第一个可以使得在某些架构上（如amd64/386）指令更加紧凑，以及减少在其他架构上计算偏移量的指令数。

不得不感叹这些大佬写的代码，真细啊~

sync.Once只有一个公共方法Do ，一个Once实例可以保证，Do方法入参中的函数f只被调用一次，多次使用Do方法执行f函数，只有第一次调用会被执行。常用于延迟初始化等。

func (o *Once) Do(f func()) {
	// Note: Here is an incorrect implementation of Do:
	//
	//	if atomic.CompareAndSwapUint32(&o.done, 0, 1) {
	//		f()
	//	}
	//
	// Do guarantees that when it returns, f has finished.
	// This implementation would not implement that guarantee:
	// given two simultaneous calls, the winner of the cas would
	// call f, and the second would return immediately, without
	// waiting for the first's call to f to complete.
	// This is why the slow path falls back to a mutex, and why
	// the atomic.StoreUint32 must be delayed until after f returns.

	if atomic.LoadUint32(&o.done) == 0 {
		o.doSlow(f)
	}
}

func (o *Once) doSlow(f func()) {
	o.m.Lock()
	defer o.m.Unlock()
	if o.done == 0 {
		defer atomic.StoreUint32(&o.done, 1)
		f()
	}
}

Do方法使用done来标识当前Once实例是否被执行过，仅当首次执行的时候才会调用doSlow，doSlow中的逻辑比较容易理解，使用sync.Mutex进行加锁，执行f，done置为1。逻辑看着很简单，但实际上这几行代码还是比较精妙的，让我们来思考几个问题：

  • 为什么需要在Do中判断一遍done的值，doSlow里再判断一遍，而不是只需要doSlow里面的这段逻辑？
  • 为什么需要先执行f，再更新done的值，而不是反过来？

针对第一个问题，还是比较容易想到的，如果没有外层的atomic.LoadUint32(&o.done) == 0，直接使用doSlow中的逻辑，会导致每一个Do的调用都会竞争互斥锁Mutex，这样会严重影响性能，外层加一个对done的判断，可以避免大量的锁竞争，当Do(f)已经被执行过后，后续的Do调用都可以迅速的返回。

if atomic.LoadUint32(&o.done) == 0 {
    o.doSlow(f)
}

第二个问题就需要稍微细致地去思考下，执行顺序反过来会有什么影响了，源码注释中解释了这个问题。

Note: Here is an incorrect implementation of Do:

if atomic.CompareAndSwapUint32(&o.done, 0, 1) {
	f()
}

注释中给出了一种Do的错误实现，直接使用CAS来更新done的值，这样可以保证done只被成功更新一次，f也就自然只会被执行一次了，这样看起来好像没问题？是的，这确实能保证f只能被执行一次，但问题在于当Do执行完毕返回时，f还没有完成，试想一个这样的场景：

  • A调用Do，同时B调用了Do

  • A赢得了CAS的执行，这时候done被置为1，f正在执行中

  • B中的Do竞争CAS失败，没有等待A执行f完毕就直接返回了

如果在延时初始化的场景，B就会任务初始化完成了，实际上初始化函数f还在执行中，这样可能会导致空指针等问题发生。

Do guarantees that when it returns, f has finished.This implementation would not implement that guarantee.

意思为Do确保当它返回时，f函数已经被执行完毕了，而上面那种错误的实现保证不了这一点

同时注释中也解释了我们前面提到的第二个问题：为什么需要先执行f，再更新done的值，而不是反过来？

This is why the slow path falls back to a mutex, and why the atomic.StoreUint32 must be delayed until after f returns.

这就是为什么slowDo需要使用互斥锁(等待执行f函数的Do返回)，以及为什么atomic.StoreUint32必须在f执行完成后执行
